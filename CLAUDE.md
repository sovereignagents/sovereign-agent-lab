# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Project Is

A 5-week course framework (Nebius Academy) for building two contrasting autonomous AI agent architectures that solve the same real-world problem (Edinburgh pub booking for 160 guests). Week 1 is the current focus.

The two agent types:
- **Headless Automator** (`sovereign_agent/`) — LangGraph ReAct loop with Llama 3.3 70B via Nebius; autonomous tool selection
- **Digital Employee** (`exercise3_rasa/`) — Rasa Pro CALM; deterministic YAML-defined flows, no improvisation

## Environment Setup

The project uses two separate Python environments:

```bash
make install        # Main env (Python 3.14 via uv)
make install-rasa   # Rasa env (Python 3.10 — strict Rasa Pro requirement)
make smoke          # Verify NEBIUS_KEY
```

Required env vars in `.env` (no quotes, no spaces around `=`):
- `NEBIUS_KEY` — Llama 70B / DeepSeek R1 inference at studio.nebius.ai
- `RASA_PRO_LICENSE` — Free Developer Edition key

## Architecture

### LangGraph Agent (`sovereign_agent/`)

- `agents/research_agent.py` — `run_research_agent(task, max_turns)` using `create_react_agent`
- `tools/venue_tools.py` — 4 `@tool`-decorated LangChain tools: `check_pub_availability`, `get_edinburgh_weather`, `calculate_catering_cost`, `generate_event_flyer`
- `tools/mcp_venue_server.py` — MCP server (stdio transport) exposing `search_venues(min_capacity, requires_vegan)`

The venue database is hardcoded in both `venue_tools.py` and `mcp_venue_server.py` — intentional for Week 1. Week 2 replaces it with Tavily web search.

### Rasa CALM Agent (`exercise3_rasa/`)

Requires Python 3.10 and its own venv. Key files:
- `data/flows.yml` — Two CALM flows: `confirm_booking` + `handle_out_of_scope`
- `domain.yml` — Slots with `from_llm` extraction (guest_count, vegan_count, deposit_amount_gbp)
- `actions/actions.py` — `action_validate_booking` business logic

Rasa uses LLM for NLU routing and slot extraction — no training data needed, just descriptive `description:` fields in flows.

### Exercise Scripts (`week1/`)

- `exercise1_context.py` — Benchmarks 3 prompt formats × 2 difficulty levels
- `exercise2_langgraph.py` — Runs Tasks A–D against the research agent
- `exercise4_mcp_client.py` — Connects to `mcp_venue_server` via stdio
- `grade.py` — Mechanical checks: JSON outputs exist, modules import, stub implementations replaced
- `outputs/*.json` — Generated by exercises (committed for grading)
- `answers/*.py` — Student reasoning answers (4 files, must be filled in)

## Grading Structure

Three layers — `make grade` only covers Layer 1:
1. **Mechanical (30pts)**: outputs exist, modules import, stubs replaced, answers filled
2. **Behavioral (40pts)**: hidden test scenarios (failure recovery, deposit enforcement, MCP filtering)
3. **Reasoning (30pts)**: essay answers in `week1/answers/`

## Code Style

- `ruff` with line-length 100, target Python 3.12
- Rasa directory excluded from ruff

## 5-Week Progression

Each week extends — not replaces — existing code:
- **Week 2**: Real web search (Tavily) + file ops + Rasa MCP access
- **Week 3**: Planner/Executor split (DeepSeek R1 + Llama 70B)
- **Week 4**: CLAUDE.md memory + vector store RAG
- **Week 5**: Observability, safety, voice pipeline (Whisper→Rasa→TTS)
